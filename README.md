## Автоматизируемые сценарии

### Сценарии перехода к формам записи

#### Через главную страницу netology.ru:
1. **Через поиск на главной странице**
   - Открыть главную страницу netology.ru
   - Найти поле поиска
   - Ввести "инженер по тестированию"
   - Выбрать нужный курс из результатов поиска
   - Перейти на страницу курса

2. **Через раздел "Программирование" на главной странице**
   - Открыть главную страницу netology.ru
   - Пролистать до раздела "Направления обучения"
   - Выбрать "Программирование"
   - Найти курс "Инженер по тестированию"
   - Кликнуть на курс для перехода

3. **Через меню "Каталог курсов"**
   - Открыть главную страницу netology.ru
   - Кликнуть "Каталог курсов" в header
   - Выбрать категорию "Программирование"
   - Найти и выбрать "Инженер по тестированию"

#### На странице курса (https://netology.ru/programs/qa-middle):
4. **Через форму внизу страницы**
   - Открыть страницу курса
   - Пролистать до конца страницы
   - Найти форму записи

5. **Через кнопку "Записаться"**
   - Открыть страницу курса
   - Найти кнопку "Записаться" (обычно в header или в верхней части страницы)
   - Кликнуть на кнопку для открытия формы

### Сценарии работы с формами

#### Форма "Получить консультацию" (из раздела программирования):
6. **Успешное заполнение формы консультации**
   - Заполнить поле "Имя" валидными данными
   - Заполнить поле "Телефон" валидным номером
   - Нажать кнопку "Получить консультацию"

7. **Валидация формы консультации**
   - Проверка обязательности поля "Имя"
   - Проверка обязательности поля "Телефон"
   - Проверка формата телефона
   - Проверка граничных значений для имени

#### Форма записи на курс (со страницы профессии):
8. **Успешное заполнение формы записи**
   - Заполнить поле "Имя" валидными данными
   - Заполнить поле "Телефон" валидным номером
   - Нажать кнопку "Записаться"

9. **Валидация формы записи**
   - Проверка обязательности всех полей
   - Валидация формата телефона
   - Проверка различных вариантов ввода имени

### Негативные сценарии:
10. **Заполнение невалидными данными**
    - Неправильный формат телефона
    - Специальные символы в имени
    - Пустые обязательные поля

## Инструменты автоматизации

### Язык программирования:
- **Java 11** - стабильная LTS-версия, широкий набор библиотек для тестирования

### Фреймворк для автоматизации веб-UI:
- **Selenium WebDriver** - промышленный стандарт для веб-автоматизации
- **Selenide** - упрощает работу с Selenium, встроенные умные ожидания

### Фреймворк для тестирования:
- **JUnit 5** - современный фреймворк, поддерживает параметризованные тесты
- **AssertJ** - fluent assertions для читаемых проверок

### Система сборки:
- **Gradle** - гибкая конфигурация, высокая производительность

### Дополнительные библиотеки:
- **Faker** - генерация реалистичных тестовых данных
- **Allure Framework** - создание детальных отчетов о тестировании
- **WebDriverManager** - автоматическое управление драйверами браузеров

### Браузер для тестирования:
- **Google Chrome** - наиболее популярный браузер среди пользователей

## Необходимые разрешения и доступ

### Юридические разрешения:
- Официальное письменное разрешение от ООО "Нетология" на проведение автоматизированного тестирования
- Согласование времени тестирования для минимизации impact на продакшн-среду

### Тестовые данные:

#### Валидные данные:
- **Имена:** Кириллица ("Иван Петров"), латиница ("John Smith"), граничные значения
- **Телефоны:** Российские форматы (+7 (999) 123-45-67)

#### Невалидные данные:
- **Имена:** пустая строка, только пробелы, спецсимволы, цифры
- **Телефоны:** буквы, недостаточное количество цифр

### Технический доступ:
- Доступ к тестовой среде (если доступна) для безопасного тестирования
- Разрешение на установку ПО для настройки тестового окружения

## Возможные проблемы

### Технические проблемы:
1. **Изменение структуры DOM и локаторов**
   - Регулярные обновления UI могут ломать XPath/CSS-локаторы
   - **Решение:** Использование стабильных локаторов, стратегия обновления тестов

2. **Динамическая загрузка контента**
   - Элементы могут подгружаться асинхронно после загрузки страницы
   - **Решение:** Реализация явных ожиданий (Explicit Waits)

3. **Popup-окна и модальные окна**
   - Всплывающие окна могут перекрывать элементы
   - **Решение:** Ожидание появления/исчезновения окон

### Организационные проблемы:
4. **Отсутствие выделенной тестовой среды**
   - Тестирование приходится проводить на продакшн-среде
   - **Решение:** Тестирование в нерабочие часы, использование тестовых данных

5. **Ограничения по нагрузке**
   - Частые запросы могут привести к блокировке IP
   - **Решение:** Добавление пауз между тестами

### Проблемы с данными:
6. **Конфликт тестовых данных**
   - Повторное использование одних и тех же данных может вызывать ошибки
   - **Решение:** Генерация уникальных данных для каждого прогона тестов

## Команда проекта

### Основная команда:

**Инженер по автоматизации тестирования** (1 человек)
- Разработка и поддержка фреймворка автоматизации
- Написание и выполнение тестовых сценариев
- Анализ результатов тестирования

**Ручной тестировщик** (1 человек)
- Валидация результатов автоматических тестов
- Баг-репортинг
- Подготовка тестовых данных

### Консультационная поддержка:

**Frontend-разработчик** (0.5 человека, частичная занятость)
- Консультации по структуре DOM
- Помощь в создании стабильных локаторов
